# Data Model: End-to-End Order Processing Flow

**Feature**: `001-order-checkout-flow`
**Date**: 2026-01-28
**Phase**: 1 — Design & Contracts

> **Key Finding**: All entities already exist in the backend schema. NO new models, NO migrations required. This document captures the existing entities relevant to the order flow and the critical ID mapping between frontend and backend.

## Entities

### Order (`backend/Models/Order.cs`)

Represents a confirmed customer purchase.

| Field | Type | Constraints | Notes |
|-------|------|------------|-------|
| `OrderId` | `int` (PK) | Auto-increment | Internal identifier |
| `OrderNumber` | `string` | Required, MaxLength(50) | Format: `ORD-YYYY-XXXXX`, generated by backend |
| `TotalAmount` | `decimal(18,2)` | Required, > 0.01 | Items subtotal only (no tax/shipping) |
| `Status` | `string` | Required, MaxLength(50), default `"Pending"` | Values: Pending, Processing, Shipped, Delivered, Cancelled |
| `ShippingAddress` | `string` | Required | JSON-serialized `AddressDto` |
| `CustomerEmail` | `string?` | MaxLength(200) | Optional |
| `CustomerName` | `string?` | MaxLength(200) | Optional |
| `CreatedAt` | `DateTime` | Default `DateTime.UtcNow` | |
| `OrderItems` | `ICollection<OrderItem>` | Navigation property | One-to-many |

## Relationships

```text
Order 1───*  OrderItem *───1  Product
```

- An **Order** has one or more **OrderItems** (cascade delete via EF Core convention).
- Each **OrderItem** references exactly one **Product** (foreign key).
- **ShippingAddress** is embedded as a JSON string within **Order**, not a separate table.

## ID Mapping: Frontend ↔ Backend

This is the critical data flow for product identification across the boundary:

```text
Backend DB                 Frontend API Layer               Frontend Cart/Checkout
─────────────             ───────────────────              ──────────────────────
Product.ProductId (int)   mapBackendProduct(dto):          Product.id (string)
       42           ──→     id: dto.productId.toString()      "42"
                            ──→ stored in cart as            ──→ parseInt(product.id)
                                productId: product.id             ──→ 42 (back to int)
```

| Layer | Field | Type | Example |
|-------|-------|------|---------|
| Backend DB | `Product.ProductId` | `int` | `42` |
| API Response | `BackendProductDto.productId` | `number` | `42` |
| Frontend mapping | `mapBackendProduct()` → `Product.id` | `string` | `"42"` |
| Cart storage | `CartItem.productId` | `string` | `"42"` |
| Order request build | `parseInt(product.id)` | `number` | `42` |
| Backend request | `BackendOrderItemRequest.productId` | `int` | `42` |

**Implication**: No slug lookup is needed at checkout. The `Product.id` in the frontend already IS the backend integer ID in string form, provided the product was loaded via the API (which checkout's `getProductById()` ensures).

## Validation Rules (Backend-Enforced)

These exist in the backend and do NOT need frontend reimplementation. The frontend sends data and handles error responses.

| Rule | Location | Error Behavior |
|------|----------|----------------|
| All product IDs must exist in DB | `OrdersController.CreateOrder` | 400: `"Products not found: [ids]"` |
| Total must match within $0.01 | `OrdersController.CreateOrder` | 400: `"Total amount mismatch"` |
| Items list non-empty | `CreateOrderRequest` model validation | 400: standard model validation |
| Email format valid | `CreateOrderRequest` model validation | 400: standard model validation |
| Customer name >= 2 chars | `CreateOrderRequest` model validation | 400: standard model validation |
| Quantity >= 1, UnitPrice > 0 | `OrderItemRequest` model validation | 400: standard model validation |

## State Transitions

```text
Order.Status flow (backend-managed):

  [Created] ──→ "Pending"
                    │
                    ▼
               "Processing"  (future: when payment confirmed)
                    │
                    ▼
                "Shipped"    (future: when shipment created)
                    │
                    ▼
               "Delivered"   (future: when delivered)

               "Cancelled"   (future: user/admin cancellation)
```

> For P1/P2, all orders are created with status `"Pending"`. Status transitions are out of scope.

## Changes Required

**None.** All entities, relationships, validations, and DTOs are pre-existing and correct. The implementation work is entirely in the frontend layer wiring to these existing structures.
